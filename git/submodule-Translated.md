## 1. مقدمه: ساب‌ماژول چیست؟

وقتی روی یک پروژه کار می‌کنید، گاهی لازم می‌شود از کد یک ریپوزیتوری دیگر داخل پروژه‌ی خودتان استفاده کنید. مثلاً فرض کنید در حال ساخت یک وب‌سایت هستید و می‌خواهید از یک کتابخانه‌ی آماده‌ی CSS یا یک قالب Bootstrap که در ریپو جداگانه‌ای قرار دارد استفاده کنید. یکی از راه‌ها این است که فایل‌های آن را کپی کنید داخل پروژه‌ی خودتان؛ اما این کار دو مشکل بزرگ دارد: اولاً وقتی ریپوی کتابخانه آپدیت شد، شما باید دوباره دستی کپی کنید و دوم اینکه تاریخچه‌ی گیت (git history) دو ریپو با هم قاطی می‌شود و پیگیری تغییرات سخت می‌شود.

**Git Submodule** دقیقاً برای حل همین مشکل ساخته شده است. ساب‌ماژول به زبان ساده یعنی یک ریپوزیتوری گیت که داخل ریپوزیتوری دیگر قرار گرفته، اما تاریخچه‌ی مستقل خودش را حفظ می‌کند. به عبارت دیگر، ریپوی اصلی فقط یک «اشاره» (reference) به یک commit خاص از ریپوی فرعی نگه می‌دارد، نه خود فایل‌ها را. این یعنی می‌توانید نسخه‌ی خاصی از یک کتابخانه را داخل پروژه‌ی خودتان pin کنید و هر زمان که خواستید، آن را به نسخه‌ی جدیدتر ارتقا دهید، بدون اینکه تاریخچه‌ها با هم ترکیب شوند.

### چه زمانی از Submodule استفاده کنیم؟

- وقتی از یک کتابخانه یا ماژول مشترک بین چند پروژه استفاده می‌کنید و می‌خواهید نسخه‌ی آن را در هر پروژه مستقل نگه دارید.
- وقتی کدی از تیم یا سازمان دیگری را استفاده می‌کنید و نمی‌خواهید آن را fork یا کپی کنید.
- وقتی یک پروژه‌ی بزرگ را به چند ریپوی کوچک‌تر تقسیم کرده‌اید ولی می‌خواهید یک «سوپر-پروژه» (super-project) داشته باشید که همه را کنار هم ببیند.

### چه زمانی **نباید** از Submodule استفاده کنیم؟

- وقتی تیم شما با گیت آشنایی کم‌عمقی دارد، چون ساب‌ماژول‌ها منحنی یادگیری دارند و کار اشتباه می‌تواند گیج‌کننده شود.
- وقتی کد فرعی به‌طور مداوم و در همان لحظه تغییر می‌کند و نیاز به sync زنده دارید (در این حالت monorepo بهتر است).
- وقتی فقط می‌خواهید چند فایل را از ریپوی دیگر بردارید و در پروژه‌ی خودتان تغییر دهید (در این حالت کپی ساده‌تر است).

## 2. پیش‌نیازها

قبل از شروع، مطمئن شوید موارد زیر را دارید:

1. **گیت نصب شده باشد** — حداقل نسخه‌ی 2.18 توصیه می‌شود (نسخه‌های جدیدتر از پروتکل HTTPS و SSH بهتری پشتیبانی می‌کنند). برای بررسی نسخه:

   ```bash
   git --version
   ```

2. **حساب GitHub** و دسترسی به ریپوهایی که می‌خواهید به عنوان ساب‌ماژول اضافه کنید (عمومی یا خصوصی).
3. **آشنایی پایه با گیت**: می‌دانید `clone`، `commit`، `push` و `branch` چه کار می‌کنند. اگر نه، اول [این آموزش رسمی](https://docs.github.com/en/get-started/getting-started-with-git) را بخوانید.
4. **تنظیم SSH یا HTTPS برای GitHub**: اگر ریپو خصوصی است، بهتر است SSH key تنظیم کنید تا هر بار پسورد نخواهد شد.

> 💡 **نکته:** اگر تازه با گیت شروع کرده‌اید، پیشنهاد می‌کنم قبل از ادامه، حتماً مفهوم `commit` و `working directory` را خوب درک کنید، چون ساب‌ماژول‌ها روی این دو مفهوم ساخته شده‌اند.

## 3. مفاهیم پایه و اصطلاحات

برای اینکه بقیه‌ی راهنما را راحت متوجه شوید، چند اصطلاح را با هم مرور می‌کنیم:

| اصطلاح | معنی |
|---|---|
| **Super-project (ریپوی اصلی/پدر)** | ریپویی که ساب‌ماژول داخل آن قرار دارد. مثلاً پروژه‌ی اصلی شما. |
| **Submodule (ساب‌ماژول/ریپوی فرعی)** | ریپویی که به عنوان زیرمجموعه داخل super-project قرار گرفته. |
| **`.gitmodules`** | فایلی در ریشه‌ی super-project که اطلاعات همه‌ی ساب‌ماژول‌ها (آدرس و مسیر محلی) را نگه می‌دارد. این فایل باید commit شود. |
| **Detached HEAD** | حالتی که در آن ساب‌ماژول به جای یک branch، روی یک commit خاص «سوار» شده است. این حالت پیش‌فرض ساب‌ماژول‌هاست. |
| **Pin کردن** | قفل کردن ساب‌ماژول روی یک commit خاص، طوری که super-project همیشه همان نسخه را ببیند تا زمانی که شما صریحاً آن را آپدیت کنید. |

### ساختار داخلی

وقتی یک ساب‌ماژول اضافه می‌کنید، گیت دو چیز می‌سازد:

1. یک فایل متنی به نام `.gitmodules` در ریشه‌ی پروژه.
2. یک پوشه برای ساب‌ماژول (مثلاً `libs/my-lib/`) که داخلش یک مخفی‌گاه `.git` (نه پوشه، بلکه یک فایل که به `.git/modules/...` در super-project اشاره می‌کند) قرار دارد.

همچنین داخل پوشه‌ی `.git/modules/` ریپوی اصلی، یک نسخه‌ی کامل از git data ساب‌ماژول ذخیره می‌شود. این یعنی ساب‌ماژول یک ریپوی کامل و مستقل است، فقط به‌صورت فیزیکی داخل پوشه‌ی super-project ساکن شده.

## 4. افزودن یک ساب‌ماژول

فرض کنید یک پروژه‌ی اصلی دارید به نام `my-app` و می‌خواهید کتابخانه‌ای به نام `awesome-lib` را از GitHub به عنوان ساب‌ماژول داخل پوشه‌ی `libs/` اضافه کنید.

### مرحله ۱: به پوشه‌ی پروژه بروید

```bash
cd path/to/my-app
```

### مرحله ۲: ساب‌ماژول را اضافه کنید

```bash
git submodule add https://github.com/user/awesome-lib.git libs/awesome-lib
```

این دستور چند کار همزمان انجام می‌دهد:

1. ریپوی `awesome-lib` را کلون می‌کند داخل `libs/awesome-lib`.
2. فایل `.gitmodules` را می‌سازد (یا اگر هست، آن را به‌روزرسانی می‌کند).
3. یک ورودی در ناحیه‌ی staging اضافه می‌کند (یعنی `git add` خودکار انجام می‌دهد).

### مرحله ۳: وضعیت را ببینید

```bash
git status
```

خروجی چیزی شبیه این می‌شود:

```
Changes to be committed:
  (use "git rm --cached <file>..." to unstage)
        new file:   .gitmodules
        new file:   libs/awesome-lib
```

توجه کنید که `libs/awesome-lib` به‌صورت یک «tree entry» (اشاره به یک commit) ثبت می‌شود، نه به‌صورت فایل‌های داخلش. این دقیقاً همان مزیت ساب‌ماژول است: گیت تاریخچه‌ی آن پوشه را جدا نگه می‌دارد.

### مرحله ۴: commit کنید

```bash
git commit -m "feat: add awesome-lib as submodule"
```

### مرحله ۵: push کنید

```bash
git push origin main
```

تمام! حالا پروژه‌ی شما یک ساب‌ماژول دارد. هر کسی که بعداً این ریپو را کلون کند، باید یک مرحله‌ی اضافی انجام دهد تا محتوای ساب‌ماژول هم دانلود شود (در بخش بعد توضیح می‌دهیم).

> ⚠️ **هشدار:** اگر آدرس ساب‌ماژول را با HTTPS اضافه کنید، همه‌ی اعضای تیم می‌توانند آن را clone کنند (به‌شرط عمومی بودن ریپو). اما اگر از SSH استفاده کنید (`git@github.com:user/awesome-lib.git`)، اعضای تیم باید SSH key معتبر داشته باشند. در محیط‌های سازمانی، HTTPS معمولاً ساده‌تر است.

## 5. کلون کردن یک ریپو که ساب‌ماژول دارد

وقتی شخص دیگری (یا خودتان روی ماشین دیگر) ریپو را کلون می‌کند، به‌صورت پیش‌فرض **پوشه‌های ساب‌ماژول خالی می‌مانند**. این یکی از اولین سردرگمی‌های رایج است.

### روش اول: کلون با پرچم `--recurse-submodules` (روش پیشنهادی)

```bash
git clone --recurse-submodules https://github.com/user/my-app.git
```

این دستور پس از کلون کردن super-project، به‌صورت خودکار تمام ساب‌ماژول‌ها را هم init و update می‌کند.

### روش دوم: کلون معمولی و سپس init/update

اگر یادتان رفت `--recurse-submodules` بزنید یا ریپو را قبلاً کلون کرده‌اید:

```bash
git clone https://github.com/user/my-app.git
cd my-app

# مرحله 1: فایل .gitmodules را بخوان و تنظیمات محلی را بساز
git submodule init

# مرحله 2: محتوای هر ساب‌ماژول را دانلود کن
git submodule update
```

### روش میان‌بر: هر دو با هم

```bash
git submodule update --init --recursive
```

پرچم `--recursive` برای زمانی است که خود ساب‌ماژول‌ها هم ساب‌ماژول دارند (ساختار تودرتو).

> 💡 **نکته‌ی کاربردی:** بسیاری از تیم‌ها یک اسکریپت `setup.sh` در ریپو می‌گذارند که این دستور را اجرا کند تا اعضای جدید تیم دردسر نکشند.

## 6. مشاهده وضعیت ساب‌ماژول‌ها

برای اینکه ببینید هر ساب‌ماژول روی چه commit ای است و آیا تغییراتی داشته یا نه:

```bash
git submodule status
```

خروجی نمونه:

```
+e3b9c2a4f5d6e7f8a9b0c1d2e3f4a5b6c7d8e9f0 libs/awesome-lib (v1.2.0)
 1a2b3c4d5e6f7a8b9c0d1e2f3a4b5c6d7e8f9a0b libs/another-lib (heads/main)
```

- علامت `+` در ابتدا یعنی ساب‌ماژول روی commit دیگری است نسبت به آنچه super-project انتظار دارد (یعنی داخل ساب‌ماژول تغییر کرده یا checkout کرده‌اید).
- علامت `-` یعنی ساب‌ماژول اصلاً init نشده (پوشه خالی است).
- علامت `U` یعنی conflict وجود دارد.

### مشاهده تغییرات داخل ساب‌ماژول

```bash
cd libs/awesome-lib
git status
git log --oneline -5
```

می‌توانید داخل ساب‌ماژول دقیقاً مثل یک ریپوی معمولی کار کنید.

## 7. ویرایش، commit و push داخل ساب‌ماژول

یکی از مهم‌ترین نکته‌ها: **ساب‌ماژول یک ریپوی کامل است**. پس برای commit دادن تغییرات داخلش، باید اول داخلش commit بزنید و سپس به super-project بگویید «به این commit جدید اشاره کن».

### مرحله ۱: تغییرات را داخل ساب‌ماژول اعمال کنید

```bash
cd libs/awesome-lib

# یک فایل را ویرایش کنید
echo "patch" >> some-file.txt

git add some-file.txt
git commit -m "fix: small patch in awesome-lib"
```

### مرحله ۲: تغییرات را به ریپوی ساب‌ماژول push کنید

```bash
git push origin main
```

> ⚠️ **نکته‌ی مهم:** اگر دسترسی push به ریپوی ساب‌ماژول ندارید (مثلاً ریپوی شخص ثالث است)، اول باید آن را fork کنید و آدرس ساب‌ماژول را به fork خودتان تغییر دهید. در غیر این صورت push با خطای permission مواجه می‌شود.

### مرحله ۳: به super-project برگردید و اشاره‌گر را به‌روز کنید

```bash
cd ../..
git status
```

خروجی نشان می‌دهد که `libs/awesome-lib` تغییر کرده:

```
modified:   libs/awesome-lib (new commits)
```

حالا این تغییر را در super-project هم commit کنید:

```bash
git add libs/awesome-lib
git commit -m "chore: bump awesome-lib to latest commit"
git push origin main
```

### ترتیب بسیار مهم است!

اگر داخل ساب‌ماژول commit بزنید ولی به super-project نگویید، نفر بعدی که کلون می‌کند نسخه‌ی قدیمی را می‌بیند. همیشه این ترتیب را رعایت کنید:

1. commit داخل ساب‌ماژول
2. push ساب‌ماژول
3. commit داخل super-project (برای ثبت اشاره‌گر جدید)
4. push super-project

## 8. به‌روزرسانی ساب‌ماژول از ریپوی اصلی (parent)

وقتی ریپوی ساب‌ماژول (مثلاً `awesome-lib`) در GitHub آپدیت می‌شود (تیم آن کتابخانه release جدیدی منتشر کرده)، super-project شما به‌صورت خودکار نمی‌فهمد. باید دستی آن را به‌روز کنید.

### روش اول: آپدیت همه‌ی ساب‌ماژول‌ها با هم

```bash
git submodule update --remote
```

این دستور آخرین commit از branch پیش‌فرض (معمولاً `main`) هر ساب‌ماژول را می‌گیرد و working directory را به آن می‌برد. اما هنوز در super-project ثبت نشده؛ باید commit بزنید:

```bash
git add libs/awesome-lib
git commit -m "chore: update awesome-lib to latest upstream"
git push origin main
```

### روش دوم: آپدیت فقط یک ساب‌ماژول خاص

```bash
git submodule update --remote libs/awesome-lib
```

### روش سوم: آپدیت به branch خاص (نه پیش‌فرض)

اگر می‌خواهید ساب‌ماژول را روی branch دیگری (مثلاً `develop`) دنبال کنید:

```bash
git config -f .gitmodules submodule.libs/awesome-lib.branch develop
git submodule update --remote
```

این تغییر `.gitmodules` را هم تغییر می‌دهد، پس باید commit کنید تا برای بقیه‌ی تیم هم اعمال شود.

> 💡 **نکته:** بعد از `update --remote`، ساب‌ماژول در حالت **detached HEAD** قرار می‌گیرد (یعنی روی یک commit خاص است، نه روی branch). اگر می‌خواهید روی branch بمانید و تغییر بدهید، باید دستی checkout کنید (بخش بعد).

## 9. کار با Branch و Tag در ساب‌ماژول

### رفتن به یک branch خاص داخل ساب‌ماژول

```bash
cd libs/awesome-lib
git checkout main
```

حالا می‌توانید تغییر بدهید و commit بزنید، چون روی branch هستید (نه detached).

### Pin کردن ساب‌ماژول به یک Tag خاص

فرض کنید `awesome-lib` نسخه‌ی `v2.0.0` را منتشر کرده و می‌خواهید پروژه دقیقاً روی همین نسخه بماند:

```bash
cd libs/awesome-lib
git fetch --tags
git checkout v2.0.0
cd ../..
git add libs/awesome-lib
git commit -m "chore: pin awesome-lib to v2.0.0"
```

از این به بعد، هر کسی که super-project را کلون کند، دقیقاً نسخه‌ی `v2.0.0` از `awesome-lib` را می‌بیند — حتی اگر نسخه‌های جدیدتری منتشر شوند — تا زمانی که شما صریحاً آن را ارتقا دهید.

### رفتن به یک commit خاص (با hash)

```bash
cd libs/awesome-lib
git checkout 1a2b3c4d5e6f7a8b9c0d
cd ../..
git add libs/awesome-lib
git commit -m "chore: pin awesome-lib to specific commit"
```

این روش برای سناریوهایی مفید است که یک باگ در نسخه‌ی جدید وجود دارد و می‌خواهید به آخرین نسخه‌ی پایدار برگردید.

## 10. حذف یک ساب‌ماژول

حذف ساب‌ماژول در گیت چند مرحله‌ای است و متأسفانه یک دستور ساده‌ی `git submodule remove` وجود ندارد (حداقل تا نسخه‌های قدیمی‌تر). مراحل کامل:

### مرحله ۱: ساب‌ماژول را از index خارج کنید

```bash
git submodule deinit -f libs/awesome-lib
```

این دستور فایل‌های تنظیمات محلی (داخل `.git/config`) را پاک می‌کند.

### مرحله ۲: آن را از tree حذف کنید

```bash
git rm -f libs/awesome-lib
```

### مرحله ۳: پوشه‌ی `.git/modules` را پاک کنید

```bash
rm -rf .git/modules/libs/awesome-lib
```

> ⚠️ این مرحله را حتماً انجام دهید. در غیر این صورت، اگر بعداً ساب‌ماژولی با همان نام اضافه کنید، گیت از داده‌ی قدیمی استفاده می‌کند و خطاهای عجیبی می‌گیرید.

### مرحله ۴: commit کنید

```bash
git commit -m "chore: remove awesome-lib submodule"
```

### مرحله ۵: اگر فایل `.gitmodules` خالی شد

اگر این تنها ساب‌ماژول بود، فایل `.gitmodules` خالی می‌ماند. بهتر است آن را هم پاک کنید:

```bash
git rm .gitmodules
git commit -m "chore: remove .gitmodules"
```

## 11. جابجایی و تغییر مسیر ساب‌ماژول

### تغییر آدرس remote یک ساب‌ماژول

گاهی ریپو منتقل می‌شود (مثلاً از یک سازمان به سازمان دیگر در GitHub). برای به‌روزرسانی آدرس:

```bash
git submodule set-url libs/awesome-lib https://github.com/new-org/awesome-lib.git
```

این دستور از نسخه‌ی 2.18+ گیت موجود است. در نسخه‌های قدیمی‌تر باید دستی `.gitmodules` را ویرایش می‌کردید و سپس:

```bash
git submodule sync
git submodule update --init --recursive
```

### جابجایی فیزیکی ساب‌ماژول به مسیر جدید

گیت دستور مستقیمی برای این کار ندارد. روش پیشنهادی:

1. ساب‌ماژول را طبق بخش [10](#10-حذف-یک-ساب-ماژول) حذف کنید.
2. در مسیر جدید دوباره اضافه کنید:

   ```bash
   git submodule add https://github.com/user/awesome-lib.git new/path/awesome-lib
   ```

3. commit بزنید.

## 12. خطاهای رایج و Troubleshooting

### مشکل ۱: `fatal: 'libs/awesome-lib' already exists in the index`

**علت:** قبلاً ساب‌ماژول را اضافه کرده‌اید ولی به‌درستی حذف نشده.

**راه‌حل:**

```bash
git rm --cached libs/awesome-lib
rm -rf .git/modules/libs/awesome-lib
```

سپس دوباره `git submodule add` را اجرا کنید.

### مشکل ۲: ساب‌ماژول در حالت **detached HEAD** است

وقتی `git submodule update` اجرا می‌کنید، ساب‌ماژول روی یک commit خاص می‌رود، نه روی branch. اگر سعی کنید commit بزنید، ممکن است commit‌ها گم شوند.

**راه‌حل:** اول به branch برگردید و سپس commit‌های احتمالی را cherry-pick کنید:

```bash
cd libs/awesome-lib
git log --oneline -3   # commit گم‌شده را پیدا کنید (اگر هست)
git checkout main
git cherry-pick <commit-hash>   # اگر commit‌ای در حالت detached داده‌اید
```

### مشکل ۳: `error: Your local changes to the following files would be overwritten by checkout`

**علت:** داخل ساب‌ماژول تغییرات commit‌نشده دارید و گیت می‌خواهد آن را به commit دیگری ببرد.

**راه‌حل:** اول یا commit بزنید یا stash کنید:

```bash
cd libs/awesome-lib
git stash
# یا
git add . && git commit -m "wip"
```

سپس دستور update را دوباره اجرا کنید.

### مشکل ۴: بعد از pull کردن super-project، پوشه‌ی ساب‌ماژول خالی است

**علت:** فراموش کرده‌اید بعد از pull، `git submodule update` بزنید.

**راه‌حل:**

```bash
git submodule update --init --recursive
```

برای جلوگیری از این مشکل در آینده، می‌توانید این تنظیم را فعال کنید تا گیت بعد از هر pull خودش این کار را کند:

```bash
git config --global submodule.recurse true
```

### مشکل ۵: conflict در ساب‌ماژول هنگام merge

گاهی هنگام merge دو branch در super-project، گیت نمی‌داند روی کدام commit از ساب‌ماژول باید بماند.

**راه‌حل:**

```bash
# 1. conflict را در سطح super-project به‌صورت معمول حل کنید و یکی از commit‌ها را انتخاب کنید
git checkout --theirs libs/awesome-lib   # یا --ours
git add libs/awesome-lib

# 2. سپس داخل ساب‌ماژول، آن commit را واقعاً checkout کنید
cd libs/awesome-lib
git checkout <commit-hash>
cd ../..
git add libs/awesome-lib
git commit
```

## 13. جایگزین‌ها: مقایسه با Git Subtree

ساب‌ماژول تنها راه برای استفاده از کد یک ریپوی دیگر داخل پروژه‌ی خودتان نیست. در اینجا کوتاه با `Git Subtree` مقایسه می‌کنیم.

### Git Subtree

`subtree` کد ریپوی دیگر را داخل پروژه‌ی شما کپی می‌کند، اما تاریخچه‌ی آن را در صورت نیاز حفظ می‌کند. مزیت اصلی این است که کاربران پروژه‌ی شما نیازی به دستور خاصی ندارند — کلون معمولی کافی است.

**افزودن:**

```bash
git subtree add --prefix=libs/awesome-lib https://github.com/user/awesome-lib.git main --squash
```

**به‌روزرسانی:**

```bash
git subtree pull --prefix=libs/awesome-lib https://github.com/user/awesome-lib.git main --squash
```

### مقایسه‌ی سریع

| معیار | Submodule | Subtree |
|---|---|---|
| **شفافیت برای کاربر نهایی** | ضعیف — باید `--recurse-submodules` بداند | عالی — کلون معمولی کافی است |
| **حجم ریپو** | کم — فقط اشاره نگه می‌دارد | زیاد — کد داخل ریپو کپی می‌شود |
| **تاریخچه مستقل** | بله — کاملاً جدا | قابل ترکیب (با squash یا بدون آن) |
| **ارتقای آسان** | بله با `update --remote` | ممکن است conflict بدهد |
| **پشتیبانی از تغییر داخل فرعی** | عالی | دشوارتر (push back پیچیده است) |
| **مناسب برای** | کتابخانه‌های مشترک، ماژول‌های مستقل | یک‌بار استفاده و سپس تغییر محلی |

### چه زمانی کدام را انتخاب کنیم؟

- **Submodule را انتخاب کنید اگر:** می‌خواهید تاریخچه مستقل حفظ شود، می‌خواهید به‌راحتی به نسخه‌های جدید upstream آپدیت کنید، و تیم شما با گیت راحت است.
- **Subtree را انتخاب کنید اگر:** می‌خواهید تجربه‌ی کاربری ساده‌تر باشد، می‌خواهید کد فرعی را به‌شدت تغییر دهید و دیگر به upstream نیاز ندارید، یا می‌خواهید ریپوی عمومی را برای کاربران نهایی ساده نگه دارید.

جایگزین‌های دیگر مانند **monorepo** (همه‌چیز در یک ریپو)، **Git X-Modules**، یا ابزارهایی مثل **Google's Repo** هم برای پروژه‌های بزرگ سازمانی مناسب هستند، اما معمولاً برای پروژه‌های متوسط ضرورت ندارند.

## 14. جمع‌بندی و Best Practices

### خلاصه‌ی فلش‌کارت

- **افزودن:** `git submodule add <url> <path>`
- **کلون با ساب‌ماژول:** `git clone --recurse-submodules <url>`
- **initialize بعد از کلون معمولی:** `git submodule update --init --recursive`
- **آپدیت از upstream:** `git submodule update --remote`
- **وضعیت:** `git submodule status`
- **حذف:** `git submodule deinit -f <path>` + `git rm -f <path>` + `rm -rf .git/modules/<path>`

### بهترین شیوه‌ها (Best Practices)

1. **همیشه `.gitmodules` را commit کنید.** این فایل حیاتی است و بدون آن، اعضای تیم نمی‌توانند ساب‌ماژول را پیدا کنند.

2. **به نسخه‌ی خاص (Tag یا commit) pin کنید، نه به branch.** این کار تضمین می‌کند که build شما قابل بازتولید است. اگر می‌خواهید همیشه آخرین نسخه را داشته باشید، به‌صورت دوره‌ای (مثلاً ماهانه) به‌صورت دستی آپدیت و تست کنید.

3. **یک اسکریپت setup بسازید.** فایلی مثل `scripts/setup.sh` با محتوای:

   ```bash
   #!/bin/bash
   git submodule update --init --recursive
   ```

   تا اعضای جدید تیم یک دستور بزنند و همه‌چیز آماده شود.

4. **در CI/CD حتماً `--recurse-submodules` را اضافه کنید.** مثلاً در GitHub Actions:

   ```yaml
   steps:
     - uses: actions/checkout@v4
       with:
         submodules: recursive
   ```

5. **هرگز داخل ساب‌ماژول در حالت detached HEAD commit نزنید.** همیشه اول `git checkout main` (یا branch مناسب) بزنید.

6. **ترتیب push را رعایت کنید:** اول ساب‌ماژول، بعد super-project. اگر برعکس باشد، build آپشن super-project شکست می‌خورد چون commit جدید ساب‌ماژول هنوز در دسترس عموم نیست.

7. **اسناد خوب بنویسید.** در README پروژه‌ی خود، صراحتاً بگویید که پروژه ساب‌ماژول دارد و دستور مورد نیاز برای setup را ذکر کنید.

8. **تیم را آموزش دهید.** ساب‌ماژول‌ها برای تازه‌کارها گیج‌کننده‌اند. یک جلسه‌ی کوتاه آموزشی می‌تواند ساعت‌ها debugging جلوگیری کند.

### چه زمانی فکر کنید به جای Submodule از روش دیگری استفاده کنید؟

اگر این علائم را دیدید، شاید زمان بازنگری باشد:

- اعضای تیم دائماً از ساب‌ماژول‌ها شکایت دارند و build آن‌ها مرتب می‌شکند.
- تعداد ساب‌ماژول‌ها از حدود ۵-۱۰ تا بیشتر شده و وابستگی‌ها پیچیده شده.
- ساب‌ماژول‌ها در عمل تغییرات محلی زیادی دارند که دیگر به upstream نمی‌روند — در این حالت کپی یا subtree بهتر است.
- پروژه به نقطه‌ای رسیده که monorepo مزیت‌های مدیریتی بیشتری می‌دهد.
